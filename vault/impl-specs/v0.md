# Sweep v0 — implementation spec

> Build the spine. `sweep "<curl … | sh>"` parses the invocation, fetches the script, saves it content-addressed, executes it with original `curl | sh` semantics, and records the result in SQLite. `sweep list` reads SQLite and renders installed packages. Everything else (LLM, signals, registry, TUI, more verbs) is deferred to v1 and explicitly slotted into named seams.

**Status:** v0 is fully implemented and committed; code is the source of truth. This spec describes contracts, invariants, and design choices — the internals live in the code. v1 work slots into the seams marked below.

---

## Glossary

- **Sweep home** — `~/.sweep/` (override: `$SWEEP_HOME`). The on-disk root for everything sweep persists: the DB, fetched scripts, future config and downloaded artifacts.
- **Script store** — `~/.sweep/cache/scripts/<sha256>`. Content-addressed cache of every script sweep has fetched. Identical scripts dedupe to the same file.
- **DB** — `~/.sweep/sweep.db`. SQLite database holding `packages` and `invocations` tables.
- **Install command** — the parsed form of the user-supplied `curl | sh` line: env vars, sudo flag, shell, script args, URL. Represented by the `InstallCommand` type. (Distinct from a "subcommand" — install commands are user input; subcommands are sweep verbs like `list`.)
- **Invocation** — each time the user runs `sweep "<cmd>"`. One row in the `invocations` table, regardless of outcome. (`sweep list`, unknown verbs, and other non-install runs do not produce invocation rows.)
- **Package** — one row in the `packages` table. Materializes the moment a fetch succeeds. Carries lifecycle status (`attempting | installed | failed | uninstalled | tracked`).
- **Install** — an invocation that ended with `exitCode === 0`. Promotes the package's `status` to `installed`.
- **Slug** — a human-readable identifier for a package. v0: derived from the URL host stem. v1: registry-canonical.
- **Runner shell** — the shell that executes the script (`sh`, `bash`, `zsh`). Extracted from the right side of the pipe in the user's input.
- **Fetcher** — the tool the user typed to download the script (`curl`, `wget`). Sweep ignores it at exec time; sweep fetches itself.

## Scope

**In v0:**

- Single positional arg form: `sweep "<curl … | sh>"`.
- `sweep list`.
- Idempotent first-run setup: `ensureSweepHome()`, `ensureConfig()` (v0 stub), `ensureDb()` (lazy, called from the store layer).
- Parser: 5-field extraction (`envVars`, `sudo`, `shell`, `scriptArgs`, `url`). Refuses chained/conditional shells.
- HTTP GET of the script URL, sha256 hash, content-addressed store.
- Exec via `Bun.spawn` with script bytes on stdin (preserves `curl | sh` stdin semantics).
- Stdout/stderr passthrough (inherited from child).
- SQLite persistence via `bun:sqlite`. Single DB file at `~/.sweep/sweep.db`.

**Explicitly out of v0:**

- LLM analysis of any kind.
- Deterministic signals (WHOIS, CT, hash DBs, sigstore).
- Recursive script fetching.
- Binary fetching/hashing.
- Centralized registry (canonical slugs, URL evolution tracking, cached verdicts).
- TUI install dialog.
- Confirm-before-exec prompt (any kind).
- First-run wizard (the `ensureConfig` seam exists but is a no-op stub).
- `--help` and `--version` flags. (Unknown flags pass through and get rejected by the parser.)
- Other subcommands: `update`, `away`, `show`, `track`, `from-history`, `--auto`.
- Bare `sweep` (no-arg interactive paste mode).
- LLM-extracted parser (deferred — v0's hand-rolled parser is behind the same interface).
- Config file / env-driven settings (other than `SWEEP_HOME`).

## Architecture

### Module layout

```
src/
  index.ts                    # 2-line entry: import + call main() (build entrypoint)
  main.ts                     # orchestrator — exports main(); ensure setup, dispatch verb or install
  fs.ts                       # sweepFs singleton + ensureSweepHome()
  config.ts                   # ensureConfig() — v0 stub; v1 reads ~/.sweep/config + runs wizard
  types.ts                    # cross-module type barrel (re-exports only)

  subcommands/
    types.ts                  # Subcommand interface
    dispatch.ts               # route to a subcommand handler
    registry.ts               # commands[] — list (install is the implicit fallback, not registered)
    list.ts                   # `sweep list`

  installer/
    install.ts                # runInstall(raw) — the install orchestrator
    parse.ts                  # parseInstallCommand(raw) → InstallCommand | ParseError
    fetch.ts                  # fetchScript(url) → FetchedScript (throws FetchScriptError)
    exec.ts                   # runScript(cmd, bytes) → { exitCode }

  store/
    db.ts                     # ensureDb() — bun:sqlite handle, mkdir + migrations
    packages.ts               # packages table CRUD + lifecycle transitions
    invocations.ts            # invocations write boundary
    scripts.ts                # CAS: saveScript / readScript / scriptPath

  identity/
    naming.ts                 # v0: slugFromUrl(url). v1: registry-backed.
```

### Cross-cutting conventions

- **camelCase domain types, snake_case SQL columns.** `PackageRow`, `Invocation`, etc. expose `sourceUrl`/`currentSha256`/`tsStarted`/`installedAt` in TS. SQL columns stay `source_url`/`current_sha256`/etc. The store layer mediates at the boundary (`rowToPackage`). SQL column names never leak into call sites.
- **`Outcome` codes** stay snake_case (`ran | cancelled | errored | parse_failed | fetch_failed`) because they're persisted as string literals — changing them would be a migration. The TS `Outcome` union mirrors the on-disk values verbatim.
- **Test seams use `__name` exports.** `db.ts:__resetForTests`, `fetch.ts:__setTimeoutMs`. Convention borrowed from wrap.
- **Tests pin SWEEP_HOME via preload.** `tests/sweep-home-preload.ts` sets `process.env.SWEEP_HOME` and wipes between tests. A second preload (`spawn-inherit-preload.ts`) rewrites `stdout/stderr: "inherit"` → `"ignore"` so install-script output doesn't leak into the test reporter.

## Modules (built)

### `src/fs.ts`

`sweepFs` singleton wrapping `wrap-core/fs` for `~/.sweep/` (or `$SWEEP_HOME`). `ensureSweepHome()` is the idempotent boot mkdir — `bun:sqlite` doesn't create parent dirs, so the DB layer relies on this.

### `src/config.ts`

v0 stub returning `{}`. Stable seam for v1's first-run wizard / config-file read; callers won't change shape.

### `src/installer/parse.ts`

Hand-rolled parser. Returns `InstallCommand | ParseError` — both types live in this file and are re-exported via `src/types.ts`.

**Accepted shapes** (the parser normalizes each into an `InstallCommand`):

| Input shape | Notes |
| --- | --- |
| `curl <flags> <url> \| sh` | Baseline. |
| `wget -qO- <url> \| sh` | Fetcher is `wget`; sweep doesn't care, just extracts the URL. |
| `curl … \| bash -s -- <args>` | `-s` is dropped at exec time. `--` and following args become `scriptArgs`. |
| `curl … \| sudo bash` | `sudo === true`. |
| `sudo curl … \| bash` | `sudo === false`. Sudo on the fetcher is moot — we re-fetch ourselves. |
| `VERSION=1.4 curl … \| sh` | `envVars = { VERSION: "1.4" }`. |
| `bash -c "$(curl <url>)"` | Normalize: `shell=bash, url=<url>`. |
| `bash <(curl <url>)` | Same as above. |

**Refused shapes** (the parser returns a `ParseError` with one of these `kind` values):

| Shape | `kind` |
| --- | --- |
| `""`, whitespace only | `empty` |
| `curl … && echo done`, `curl … ; sudo …`, `curl … \|\| true` | `chain` |
| `cat /tmp/x \| sh` (no fetcher, no URL) | `no-fetcher` |
| `curl <url>` (no pipe to shell) | `no-pipe` |
| `curl \| sh` (no URL) | `no-url` |
| Anything else | `unsupported` |

**Design choices:**

- Plain string scanning + targeted regex, no shell-grammar library. When in doubt, refuse — mis-parsing a hostile fragment is worse than rejecting it.
- Quote tracking is intentionally shallow: enough to keep env-var values from false-positiving the chain check, not a real shell parser. `\` escape sequences are NOT handled.
- Env-var keys are strict POSIX (`^[A-Z_][A-Z0-9_]*$`). Lowercase keys aren't auto-promoted — they cause the env prefix to terminate, and the token after is treated as the fetcher (which then rejects).
- `sudo` on the runner sets `sudo: true`; `sudo` on the fetcher (LHS of pipe) is ignored — we re-fetch ourselves so the original `sudo curl` semantics don't carry over. `sudo` flags between `sudo` and the shell (`sudo -E bash`) are consumed but not stored.
- `raw` is the verbatim input (not trimmed) — useful for the invocation log.
- The test suite includes a real-world corpus from 16 popular installers (ollama, bun, nvm, mise, deno, rustup, starship, oh-my-zsh, fnm, pnpm, atuin, eza, just, lazygit, zoxide, pyenv). All parse without falling back to `unsupported`.

### `src/installer/fetch.ts`

`fetchScript(url)` returns a `FetchedScript` (bytes + sha256 + finalUrl + fetchedAt + status). Throws `FetchScriptError` (class with a `reason` discriminant: `non-2xx | timeout | too-large | network`) on any failure path.

**Design choices:**

- A **single** `AbortController` covers both the headers fetch AND the body read. Without that, a slow-loris server could trickle bytes past the 30s budget while the body-read had no timer. The `clearTimeout` sits in an outer `finally` so it fires regardless of which phase threw.
- 5 MiB body cap enforced twice: a `Content-Length` pre-check refuses an honest oversize server before reading anything; a post-`arrayBuffer()` length check covers servers that omit/lie about the header.
- Abort detection prefers `e.name === "AbortError"` over `signal.aborted` so a real network error in the microsecond window before the timer fires doesn't get mislabeled as a timeout.
- Non-2xx skips the body read entirely (don't waste bandwidth on error pages).
- `__setTimeoutMs(ms)` is a test-only seam — the install code never overrides it.

### `src/installer/exec.ts`

`runScript(cmd, scriptBytes)` returns `{ exitCode }`. Thin `Bun.spawn` wrapper.

**Exec recipe:** `argv = [...(cmd.sudo ? ["sudo"] : []), cmd.shell, "-s", "--", ...cmd.scriptArgs]`. `-s --` is always included — harmless when `scriptArgs` is empty, required when not (so the shell treats trailing tokens as `$1..$N` rather than as a script filename).

**Env merge:** `{ ...process.env, ...cmd.envVars }` — later spread wins, so user-supplied env vars OVERRIDE the parent process's.

**stdin semantics:** Script body arrives via stdin. The script's stdin is the byte stream, not the user's terminal — matches `curl | sh` so installers that grab `/dev/tty` for prompts (ollama, nvm) keep working.

**Failure surfacing:** Non-zero exit codes are RETURNED, not thrown. Missing-shell (`ENOENT` from `Bun.spawn`) propagates as a throw — `runInstall`'s top-level catch path is for this class of unexpected failure. No retry, no timeout, no signal interception.

### `src/installer/install.ts`

`runInstall(raw): Promise<number>` is the orchestrator. Returns the process exit code `sweep "<cmd>"` should terminate with. Sequence:

1. **Parse.** On `ParseError`: write a single `parse_failed` invocation row (no package, no url, no sha), print to stderr, return **2**.
2. **Fetch.** On any `FetchScriptError`: write a `fetch_failed` invocation row (url + parsed JSON, no sha, no package), print to stderr, return **1**.
3. **Save script** to CAS (idempotent — same sha → no-op).
4. **Resolve package** via `findOrCreatePackage({ url, slug: slugFromUrl(url) })`. New rows materialize with `status: 'attempting'`.
5. **v1 seam:** analysis + dialog. v0 no-op.
6. **Run exec.** Capture `exitCode`.
7. **Persist outcome** atomically: `ensureDb().transaction(() => { ... })` wraps both the invocation INSERT (`ran` or `errored` per exitCode) and `updatePackageOnExec`. The package status transitions live in `store/packages.ts` (see lifecycle table below).
8. Return `exitCode`.

`tsStarted` is captured at function entry (before parse) and threaded through every code path so error rows also have it. **Every** terminal path writes exactly one invocation row.

### `src/store/db.ts`

`ensureDb()` is the idempotent singleton accessor — opens `~/.sweep/sweep.db` with WAL + FKs and runs migrations on first call. WAL persists on disk; FKs are per-connection so we apply them every open. `__resetForTests()` is the preload's reset hook (closes the handle so the next `ensureDb()` opens against the wiped home).

### `src/store/scripts.ts`

Content-addressed bytes store at `<sweepFs.root>/cache/scripts/<sha256>`. `saveScript` uses `writeFileSync(..., { flag: "wx" })` (O_EXCL) so concurrent CAS writers race cleanly — the loser gets `EEXIST` which is success by definition (same sha → same bytes). `ENOENT` triggers a parent-mkdir + retry. `readScript` returns `null` on `ENOENT` (no precheck — fewer syscalls, no TOCTOU window). Bytes flow through `node:fs` directly because `AppFs.write` is utf-8 text only.

### `src/store/packages.ts`

Typed CRUD over the `packages` table. `PackageRow` is camelCase. `findOrCreatePackage({ url, slug })` uses `INSERT … ON CONFLICT(source_url) DO NOTHING` + follow-up `SELECT` — single-round-trip, multi-process safe via the `UNIQUE` constraint, returns existing rows untouched (slug never overwritten). `listInstalledPackages()` filters to `status='installed'` ordered by `installedAt DESC`.

`updatePackageOnExec({ packageId, sha256, exitCode, ranAt })` is **one** UPDATE with CASE expressions encoding the entire lifecycle table — no read-modify-write. Does NOT open its own transaction, so callers can wrap it together with `insertInvocation`.

**Lifecycle of a `packages` row** (v0):

| Trigger | `status` after |
| --- | --- |
| First `findOrCreatePackage` for a `sourceUrl` | `attempting` |
| `exitCode === 0` | `installed` (preserves prior `installedAt` if non-null; refreshes `currentSha256`) |
| `exitCode !== 0`, prior status `attempting` | `failed` |
| `exitCode !== 0`, prior status `installed` | unchanged (still `installed`; the new invocation row records the failure) |

`uninstalled` and `tracked` are reserved for v1 (`sweep away`, `sweep track`) — never set in v0.

### `src/store/invocations.ts`

Write boundary for the `invocations` table. `Invocation` is camelCase. `insertInvocation` is fire-and-forget — single INSERT, no SELECT (every field is known at write time). `id` defaults to `crypto.randomUUID()`; explicit id is honored. Does not open its own transaction. No `rowToInvocation` mapper yet — invocations.ts is write-only in v0; the first reader (v1) will add it.

### `src/identity/naming.ts`

`slugFromUrl(url)` derives a human-readable slug:

1. `new URL(url)`; if it throws, return `"unknown"`.
2. `hostname.toLowerCase()`. **Empty hostname → `"unknown"`** (e.g. `file:///foo` — empty slug would break list rendering).
3. Strip a single leading prefix label matching one of: `www.`, `get.`, `install.`, `download.`, `dl.`, `cdn.`. At most one strip.
4. Return the first dot-separated label (or `"unknown"` if empty).

v0 does not deduplicate slugs across packages; collisions are tolerated (the `source_url` UNIQUE constraint disambiguates rows; `sweep list` shows source URL alongside slug).

Signature is stable across v0 and v1; v1 swaps the body for a registry call.

### `src/types.ts`

Cross-module type barrel. Each module owns its types locally; this file re-exports them so callers have one import site. Re-exports: `InstallCommand`, `ParseError`, `FetchedScript` from `installer/`; `Outcome`, `Invocation` from `store/invocations.ts`; `PackageRow` from `store/packages.ts`.

### `src/subcommands/`

`types.ts` defines `Subcommand = { name, description, run(argv): Promise<number> }`. Deliberately simpler than wrap's CLIFlag/option union — v0 has one verb. `registry.ts` exports `commands: Subcommand[] = [listCmd]`; install is the implicit fallback in `main.ts`, not a registered verb. `dispatch.ts` finds by name and throws on miss (defensive — main only calls it on a match). `list.ts` reads `listInstalledPackages()` and prints one tab-separated `slug\tsourceUrl\tinstalledAt` line per row, or `"No packages installed."` on empty. Always exit 0.

### `src/main.ts`

Top-level orchestrator. Calls `ensureSweepHome()` + `ensureConfig()`, then takes `process.argv[2] ?? ""` and either dispatches it as a verb (currently only `list`) or hands it to `runInstall`. Sets `process.exitCode` rather than calling `process.exit` so in-process tests can `await main()`. Top-level try/catch is the safety net for unexpected throws — `runInstall` already handles its own parse/fetch failures and never throws on those.

Empty argv is intentional: it flows through to `parseInstallCommand("")`, which returns a `ParseError` of kind `empty`. `sweep` with no args exits 2 the same way `sweep "garbage"` does.

No `--help` / `--version`. Unknown flags fall through to the parser and get rejected.

`ensureDb` is NOT called from main — store modules lazy-open on demand, so `sweep list` and `sweep "<cmd>"` both reach the DB through the store layer.

### Wiring & entry point

`src/main.ts` exports `main()` only — no top-level invocation, no shebang. `src/index.ts` is two lines that import and call it:

```ts
import { main } from "./main.ts";
main();
```

`package.json` has `"start": "bun run src/index.ts"` and no `bin` field. Production distribution as a standalone executable via `Bun.build({ entrypoints: ["src/index.ts"], compile: { outfile: "sweep" } })` is deferred to a later release-plumbing pass (out of v0 scope). v0 is verifiable via `bun run start "<cmd>"` and `bun run start list`.

## Storage

### Filesystem

```
~/.sweep/                                 (or $SWEEP_HOME)
  sweep.db                                # SQLite, both tables
  cache/
    scripts/
      <sha256>                            # raw bytes, content-addressed
```

No JSONL log. No config file in v0.

### Schema (version 1)

Schema lives in `src/store/db.ts` (`CREATE TABLE IF NOT EXISTS …`). High level:

- `schema_meta(version INTEGER PRIMARY KEY)` — single row holding the schema version (1 in v0). PRIMARY KEY makes `INSERT OR IGNORE` idempotent on re-open.
- `packages(id, slug, source_url UNIQUE, current_sha256, status, first_seen_at, installed_at, last_ran_at)` — one row per tracked tool.
- `invocations(id, package_id → packages.id, ts_started, ts_finished, raw_input, url, sha256, install_command_json, outcome, exit_code, error_message)` — one row per `sweep "<cmd>"` run, including parse and fetch failures.

Indexes: `packages(current_sha256)`, `invocations(package_id)`, `invocations(ts_started DESC)`.

## Data flows

### `sweep "<curl … | sh>"`

```
argv ──▶ main.ts ──▶ installer/install.ts:runInstall(raw)
                                                  │
                                                  ├─▶ parse.ts          (→ ParseError → invocation, exit 2)
                                                  ├─▶ fetch.ts          (→ error → invocation, exit 1)
                                                  ├─▶ store/scripts.ts  (saveScript)
                                                  ├─▶ store/packages.ts (findOrCreatePackage)
                                                  ├─▶ identity/naming.ts
                                                  ├─▶ [v1 seam: analyze + confirm]
                                                  ├─▶ exec.ts           (run, capture exitCode)
                                                  └─▶ store/{invocations,packages}.ts (one txn)
                                                          │
                                                          └─▶ exitCode → main → process exit
```

### `sweep list`

```
argv ──▶ main.ts ──▶ subcommands/list.ts ──▶ store/packages.ts:listInstalledPackages()
                                                  │
                                                  └─▶ format + stdout
```

## Implementation order (historical)

The order followed, minimizing blocked tests: `src/fs.ts` scaffolding → `store/db.ts` → `store/scripts.ts` → `store/packages.ts` → `store/invocations.ts` → `identity/naming.ts` → `installer/parse.ts` + `types.ts` → `installer/fetch.ts` → `installer/exec.ts` + `spawn-inherit-preload.ts` → `installer/install.ts` → `subcommands/{types,dispatch,registry,list}.ts` → `main.ts` + `index.ts` + end-to-end CLI test.
